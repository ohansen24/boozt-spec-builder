This is the single reconciled plan across the six component designs (engine-api, engine-worker, supabase, frontend, validation-net, slack-ops). ## 1. Cross-cutting invariants (hold in every phase)

- **The bar never moves.** Widening the net changes candidate *supply* only. Green still = two independent GTIN-anchored `source_family` values agreeing; single family = yellow; equal-authority disagreement fails closed to red. The web UI **renders engine-assigned `status`** and must never recompute color or the green rule in the browser.
- **Provenance on every cell.** Every value carries `primary`/`secondary` `{url, method, snippet, fetched_at}` and structured `flags` — lifted out of `fv.notes` substrings at emit time, never regexed downstream.
- **Enrichment + reds-first review queue, never a "done" sheet.** Soft gates route to the queue with provenance; no gate silently ships a questionable value.
- **6-status model preserved** (`VERIFIED/ODM_SOURCED/SINGLE_SOURCE/CONFLICT/NOT_FOUND/MANUAL`) with derived color; don't collapse to 3.
- **Edge Functions are thin glue only** — never run the engine. The Render worker (service-role) does all heavy writes.

## Phase A — Engine service-ready (pure Python, no cloud)

**Goal:** turn the argv+`click.echo`+`SystemExit` CLI into a callable, side-effect-free, DI-driven engine that returns `(output_bytes, coverage_json)`, with the tiered validation net and depth knob threaded through. This is the foundation; **B/C/D cannot start their integration work until A's two contracts (the `run()` signature and the coverage JSON) are frozen.** The CLI remains as a thin shim so nothing regresses.

Sub-sequence: **A1 (seam) → A2 (net/depth)**, because A2's knobs ride on A1's `RunOpts`.

### A1 — Callable seam, coverage contract, typed exceptions
Deliverables:
- **`src/bsb/engine.py`**: move bodies of `run` (`cli.py:141-230`) + `_run_resolved` (`271-533`) + the three generic passes (`535-739`) + `_hit_ref`. Public `run(odm_bytes, brand_key, opts, *, brand_cfg, template_path, config_dir, cache_dir, rules, synonyms, prior_cells, progress, credit_budget) -> RunResult`. `RunResult.__iter__` yields `(output_bytes, coverage)` so the locked tuple contract holds literally.
- **`RunOpts`** dataclass (replaces the 15 Click options): `order_number, source_label, tier, do_resolve, do_validate, retailer_net, depth, lf_useful, allow_size_anomalies, allow_regressions, gate_mode, bases_filter`.
- **Typed exceptions** replacing every `SystemExit`/Click error: `BlockedBrandError`, `BlockingAnomalyError`, `SizeAnomalyError`, `ConflictGateError`, `EngineIngestError`, reuse `RegressionError`. Under `gate_mode="soft"` size/conflict/regression **do not raise** — recorded into `coverage["gates"]`, `outcome="needs_review"`, bytes still returned.
- **Coverage JSON v1** built in-process from in-memory `records` (never by re-parsing the Provenance sheet): `schema, order_number, brand, tier, depth, data_sheet_name, summary (RunSummary.model_dump()), fields{} per-field rollup, totals, cells[] (per-cell status/color/value/primary/secondary/notes/failure_class/flags/source_markets/agreeing_families), gates{}, credits{}, learning{}`. **This is the frozen contract C binds to** — the TS interface in the frontend design.
- **Prior-cells decoupling**: `write_output` takes injected `prior_cells` dict, writes to `BytesIO`, returns `(RunSummary, output_bytes)`. No `_read_prior_cells(out_path)` on the hot path.
- **BytesIO plumbing**: widen `parse_odm`/`parse_spec_sheet`/`_ingest_order` to accept file-like; `_odm_ref` stamps `opts.source_label` not a filesystem path.
- **Progress seam**: replace `_Progress`/`click.echo` with injected `progress(ProgressEvent)`, `_noop` default. Emit **per-master** in the LF/net loop.
- **Kill config mutation** (`_record_validator_pool` file write) → emit into `coverage.learning`. **Fix the overrides bug** (`load_order_overrides` ignoring `config_dir`, `cli.py:490`). Mount `config_dir` read-only so accidental writes fail loud.
- **CLI shim**: `cli.py` `run` becomes a thin file↔bytes wrapper passing `gate_mode="hard"`; existing golden/gate tests stay green (regression guard for the whole refactor).

### A2 — Tiered validation net + depth knob (refactor b)
Deliverables:
- **`DepthProfile`** frozen dataclass `(name, max_pages, search_limit, escalate_walls, settle_ms, scrape_wait_ms, weak_inci)`, threaded `run(opts) → passes → GenericResolver.resolve(..., depth)`. **Fix the `candidates(limit)` bug** (`generic.py:156` drops the limit). Gate wall-escalation on `depth.escalate_walls`; pass `scrape_wait_ms`/`settle_ms` through.
- **LF pre-probe**: new `LookfantasticValidator.probe(eans) -> (lf_useful, evidence)` — ≤2 masters, ≤~4 renders, no product fetch. Hoist the emptiness verdict from "after N masters" (`cli.py:398`) to "after ≤2". Route by `lf_useful`: `true`→full LF pass; `false`→skip LF entirely; `null`→probe then set tentative.
- **Always-on worldwide net**: promote the three gated passes (`_generic_validator_pass`, `_retailer_primary_pass`, `_field_completion_pass`) to always-on, unified triggering. Runs on every **non-VERIFIED** cell (red + single-source yellow), skips VERIFIED. **Fix the per-EAN spend bug** (`cli.py:571-573`) → dedup to one representative EAN per product line for name/shade/INCI; size stays per-variant (per-EAN fallback only at `deep`).
- **Resolve-capability gate fix** (`cli.py:299-304`): route `generic`/null-adapter brands (svr, aesop) into the retailer-primary engine instead of `ClickException`.
- **Market/INCI rules unchanged** — confirmed already correct (`market.py`, `pipeline.py:309-519`, `inci.py`). Only additions: `_KNOWN_HOST_MARKET` and the INCI blocklist become **table-loaded** (engine reads a dict the worker injects; CLI reads YAML). `content_only` stays observability-only (recommended).
- **CreditBudget** threaded into `GenericResolver`; on ceiling, net stops widening, remaining reds tagged `failure_class="budget_exhausted"`, `outcome="needs_review"`.

**Dependencies:** none external. **Exit criteria:** `run()` returns valid coverage JSON for a NARS order; CLI still passes goldens; a Maria-Nila-style run skips LF after ≤4 renders.

## Phase B — Supabase backend + Render worker

**Goal:** stand up the source-of-truth database, storage, thin edge glue, and the single Render background worker that claims jobs, runs `engine.run()`, streams progress, and persists artifacts + learning. **Depends on A** (callable engine + frozen coverage contract).

Sub-sequence: **B1 schema → B2 worker → B3 edge functions + reaper** (B2 and B3 can overlap once B1 lands).

### B1 — Supabase schema, RLS, seed
Deliverables:
- **Enums**: `brand_tier, adapter_kind, job_status` (per R1), `job_error_class, detection_source, request_status, credit_pass`.
- **Tables**: `profiles`, `brands` (R2; first-class `tier/dg_prerequisite/out_of_scope/lf_useful/validator_pool` + `config jsonb` + generated `runnable`), `retailer_families`, `retailer_market`, `inci_capability`, `app_settings` (holds `depth_tiers`, `credit_ceiling_default`, `google_hd`), `jobs` (the full state machine + progress + detection columns), `job_events` (audit), `credit_ledger` (per-pass), `brand_requests`. `notifications`/`audit_log`/`brand_registry_audit` deferred to D.
- **Helper fns + trigger**: `email_domain()`, `is_team_member()`, `is_admin()`, `handle_new_user` (domain-gate signups, bootstrap Oli as admin).
- **RLS**: team-wide read on jobs/brands/events/ledger; no direct client writes (all via definer RPC / edge / worker); brand_requests requester+admin scoped.
- **RPCs (SECURITY DEFINER)**: `claim_next_job(worker)` (R7), `submit_job`, `confirm_brand`, `cancel_job`, `override_regression`, `set_opts`.
- **Storage buckets** (R9), private, default-deny.
- **Realtime**: publish `jobs` (+ `brand_requests` in D).
- **`v_order_last_run` view** (R3).
- **Seed**: 12 brands from `brands.yaml`/`BRAND_ONBOARDING.md` §5 (validated: nars/benefit/maria_nila; experimental: olaplex/aesop/svr; unsupported: avene/aderma/k18/colorescience/marc_jacobs[dg]/soft_goat[oos]); `retailer_families` + `inci_capability` from `validators.yaml`; `retailer_market` seeded from `_KNOWN_HOST_MARKET`; `app_settings.depth_tiers` placeholders (R5).

### B2 — Render worker (`worker/`)
Deliverables:
- **Dockerfile** on `mcr.microsoft.com/playwright/python:v1.49.1`; bake `playwright install chromium` at build (never at boot); bundled master at `TEMPLATE_PATH`; run as non-root; `/cache` = Render persistent disk (single-attach, R6).
- **`render.yaml`**: one **Background Worker**, one Disk at `/cache`, env-group secrets (`FIRECRAWL_API_KEY`, `SUPABASE_URL`, `SERVICE_ROLE_KEY`).
- **`main.py` loop**: poll/Realtime wake → `claim_next_job` → `process`. `POLL_INTERVAL=3s`, `LEASE=15m`, `MAX_ATTEMPTS=3`, `WALL_CLOCK=90m` (§6-D8).
- **Detection pre-phase** (R10): parse ODM → `order_number` + `brand_for_order` + sheet brand-column cross-check → write `detected_brand_key/detection_source/detection_confidence/detection_signals`; agreement/user-pick → `running`; ambiguous/unknown → `awaiting_brand`; registry block → `blocked`.
- **Registry block gate** before the engine (engine raise is the backstop): `out_of_scope || dg_prerequisite || tier='unsupported'` → `blocked`. **This is the enforcement point for `dg_prerequisite`** (the engine does not enforce it).
- **`runner.py`**: rehydrate `brand_cfg` from `brands.config` (+ promoted columns + machine `lexicon`), load `prior_cells` from `v_order_last_run`'s coverage.json, call `engine.run(gate_mode="soft")`.
- **`progress.py`**: coalesced (≤1/s) writes to `jobs.progress` → Realtime; enforce wall-clock via `TimeoutAbort`.
- **Heartbeat/lease** thread + **reaper** (pg_cron or scheduled edge fn) requeuing `running` jobs past `lease_expires_at` with `attempts<MAX`.
- **Persistence**: upload output.xlsx + coverage.json to `spec-outputs`; insert per-pass `credit_ledger`; write learning back to `brands`/`retailer_market`/`inci_capability` (R13); set terminal state.
- **Failure taxonomy** → states per engine-worker §8 (transient FetchError/Playwright/Firecrawl-5xx → retry; ODM-parse/blocked/blocking-anomaly → terminal).

### B3 — Edge functions + minimal metering
Deliverables:
- **`create-job`**: verify JWT team-member, early block gate if brand pre-picked, insert `jobs` (`uploaded`), return `{job_id, signed_upload_url}`, write `job_events`.
- **`sign-download`**: visibility-check under caller JWT, sign output + coverage (TTL 1h).
- Raw Firecrawl counts into `credit_ledger` (budgets/alerts are D).

**Exit criteria:** end-to-end CLI-free run — upload a NARS xlsx via signed URL, worker claims/detects/runs/emits, `jobs` reaches `succeeded`/`needs_review`, coverage.json downloadable, `lf_useful` written back.

# Completeness Critique — BSB Self-Serve Designs

The five component designs are individually strong and each carries its own decision list. My job is the seams **between** them and the things **none** of them owns. The scariest gaps are not inside any one design — they fall in the cracks. Ranked by blast radius.

---

## Part 1 — Ranked gaps / risks (with mitigations)

### TIER A — build-blockers: contradictions and unenforced charter

**A1. "Multi-tenant" is asserted by two designs and does not exist in the schema.**
`slack-ops` and `frontend` repeatedly invoke `org_id`, "per-org monthly budget", "per-org active-job cap", "tenant isolation" — but the Supabase design has **no org/tenant concept at all**. It is single-tenant: a hard `@buyinglabs.com` gate + team-wide read of *every* job. These cannot both be true. Either (a) this is one org (Felina + team) and all the `org_id` plumbing in slack-ops §6 is dead code chasing a requirement that doesn't exist, or (b) external customers will use it, in which case **team-wide `jobs` SELECT (`jobs_read … using(is_team_member())`) is a cross-tenant data leak** and the entire RLS model is wrong.
*Mitigation:* Oli must declare the tenancy model **before any schema is written** (it changes every table's PK/RLS). If single-org: strike `org_id` everywhere and keep the domain gate. If multi-org: add `org_id` to `profiles/jobs/brand_requests/credit_ledger`, scope every RLS policy by org, and re-cost the budget tables per org.

**A2. Four incompatible `status` vocabularies across four designs.**
- Supabase: `uploaded, queued, detecting, awaiting_brand, running, needs_review, blocked, succeeded, failed, canceled`
- engine-worker: `queued, claimed, running, succeeded, needs_review, blocked, failed, timed_out`
- frontend: `queued, running, blocked, needs_review, conflict_gate, regression, error, done`
- slack-ops: `succeeded, needs_review, blocked, error`

`done` vs `succeeded`, `error` vs `failed`, `conflict_gate`/`regression` as first-class states vs folded into `needs_review`, `claimed`/`detecting`/`timed_out`/`awaiting_brand` present in some and absent in others. The frontend's whole state machine and the Realtime subscription key off this enum; a mismatch is a guaranteed runtime bug.
*Mitigation:* Freeze **one canonical `job_status` enum** (Supabase's is the most complete — add `claimed` and `timed_out` from engine-worker) in a single shared contract doc, and make `conflict_gate`/`regression` sub-reasons inside `error_class`, not top-level statuses. Same reconciliation needed for table names: `brands` (Supabase) vs `brand_registry` (everyone else), and `run_history` as a real table vs a view (Supabase FLAG-4). The engine-worker §7 contract literally reads `brand_registry` from a DB that creates `brands`.

**A3. `progress` shape mismatch — Realtime will render nothing.**
engine-worker writes a `progress` **jsonb blob** (`{stage,done,total,...}`); Supabase defines **discrete columns** (`progress_stage/current/total/message`); frontend expects a jsonb `{phase, current_master, total_masters}`. Three shapes, one column.
*Mitigation:* Pick jsonb (cheapest for the worker, matches frontend), pin the exact keys in the shared contract, and delete the discrete-column variant from the Supabase design.

**A4. Nothing enforces the "review before ship" charter — the download is an unguarded exit.**
The entire product thesis is "enrichment + reds-first review queue, never a fully-automatic done sheet." But the actual exit is a **"Download draft sheet" button that is always live**, even with 40 unreviewed reds. A non-expert (the literal target user) downloads the xlsx, uploads it to Boozt, done — the review queue was optional decoration. Once the file leaves, its provenance and red/yellow/green state are gone (Boozt ingestion strips cell fills; a user copy-pasting the Data sheet drops the Provenance sheet). There is **no review-completion state, no "acknowledge N reds" gate, no DRAFT watermark in the file itself.**
*Mitigation:* (1) Stamp the output xlsx with a visible "DRAFT — UNREVIEWED, N reds / M yellows" banner row/sheet and a `reviewed=false` flag; (2) track per-job review state (`reds_acknowledged_by`, `reds_acknowledged_at`) and only mark a job `finalized` after an explicit acknowledgement; (3) consider gating download-without-ack behind a confirm dialog that states the red count. This is the difference between the charter being real and being a slide.

**A5. The "Validation OFF" and "Deep" toggles hand a non-expert a footgun.**
Charter-wise, the most dangerous control in the UI is the one that lets Felina's team turn validation **off** ("go faster") and ship single-source, uncorroborated values — a direct charter violation, one click, one inline warning that toggles get dismissed. Symmetrically, "Deep" invites "more = better" and blows Firecrawl credits.
*Mitigation:* Remove "validation off" from the self-serve role entirely (admin-only), or require a typed confirmation + attribute it in `audit_log`. Gate "Deep" behind a confirmation showing the credit estimate, or make it admin/per-order-escalation only.

### TIER B — cost, cancellation, and the 31-min job's rough edges

**B1. Firecrawl credit ceiling ships DISABLED — the always-on worldwide net has no cap on day one.**
`app_settings.credit_ceiling_default = {"enabled":false,...}`. The validation-net design makes the generic net **always-on and worldwide**, i.e. the cost multiplier is now the default path, and the cap that would contain it is off. One "Deep" run on a many-line brand, or a retry storm (B2), can burn an unbounded number of credits before anyone notices. There is also **no enforced global monthly kill-switch** — it's described as "worker refuses to start jobs" but not built.
*Mitigation:* Ship v1 with the per-job ceiling **enabled** at a conservative default and a hard global monthly ceiling that actually blocks `claim_next_job`. Overflow → partial coverage flagged `budget_exhausted` → `needs_review` (the design already supports this outcome; just turn it on).

**B2. Retries and reaper re-run the whole 31-min job — no checkpoint, no resume.**
`MAX_ATTEMPTS=3`. A job that dies at the emit stage re-runs ingest→resolve→LF→net from scratch on retry. The shared cache saves *credits* on warm URLs but not *time* or *newly-billed searches* if cache keys shifted. A poison job (OOM/Playwright segfault) can take down the single worker, get reaped, and **crash it again** — with concurrency=1 this stalls the entire queue for up to 3× the wall-clock, and each attempt may re-bill.
*Mitigation:* (1) Cap retries harder for crash-class failures (dead-letter after 1–2, not 3); (2) treat a job that crashed the worker process as poison → straight to `failed`, don't reap-and-repeat; (3) accept no-resume for v1 but document that a retry is a full re-bill, and lean on the cache being warm; (4) monitor worker liveness (B5).

**B3. Cancellation is not wired into the engine — `cancel_job` burns the full 31 min anyway.**
Supabase defines a `cancel_job` RPC, but the engine seam and worker loop have **no cooperative cancellation**. Setting `status='canceled'` in Postgres does nothing to the running worker, which keeps rendering and billing Firecrawl for half an hour. A user who uploaded the wrong file/brand cannot stop the spend.
*Mitigation:* Have `on_progress` (which already checks the wall-clock deadline) also poll a cancel flag on the `jobs` row and raise `CancelAbort`; the worker treats it as terminal `canceled`. Cheap, reuses the existing progress seam.

**B4. Hang (not crash) is unkillable — the wall-clock is only checked inside the progress callback.**
`on_progress` enforces `WALL_CLOCK`. But if the engine hangs **inside** a synchronous Playwright/httpx call and never fires a progress event, `on_progress` never runs, the deadline is never checked, and the **heartbeat thread keeps renewing the lease forever** → the job hangs indefinitely, the reaper never fires (lease is alive), the single worker is wedged, the whole queue stalls silently.
*Mitigation:* Move the wall-clock enforcement into the independent heartbeat thread (it already runs on a timer): on each tick, if `now > deadline`, force-terminate the run (process-level `SIGALRM`/watchdog kill). Do not rely on the engine to volunteer a progress event to die.

**B5. No worker-liveness monitor — a dead worker fails silently.**
Single background worker, concurrency 1. If it crashes on boot, OOMs permanently, or Render fails to redeploy, **jobs pile up in `queued` and nobody is paged.** All the Slack alerting is per-job (fired *by* the worker) — a dead worker sends no alerts by definition. Users see "queued" forever with no ETA.
*Mitigation:* An external heartbeat: a `pg_cron`/scheduled Edge Function that checks "oldest `queued` job age" and "last worker heartbeat" and pings Oli if either crosses a threshold. This is the one alert that cannot live on the worker.

**B6. Graceful shutdown on deploy is undefined.**
Render redeploys/restarts send SIGTERM mid-job. Without a handler, the job dies and waits up to the 15-min lease before the reaper requeues it (31 min + latency lost).
*Mitigation:* Trap SIGTERM, immediately set the in-flight job back to `queued` (attempts unchanged) and exit, so it's re-claimable instantly rather than after lease expiry.

**B7. Queue position / ETA is not surfaced; concurrency=1 makes waits long and invisible.**
Ten uploads = the tenth waits ~5 hours cold, shown only as "queued." The frontend's honest "~30 min" copy is per-job and ignores queue depth.
*Mitigation:* Compute and show queue position + naive ETA (position × ~31 min cold / few-min warm) on the `queued` state; it's a trivial `count(*) where status in (queued,running) and created_at < mine`.

**B8. `awaiting_brand` interaction with a single worker is unspecified — risk of deadlock.**
Detection runs as a worker phase (both Supabase and frontend recommend worker-side). If detection is ambiguous → `awaiting_brand`, and the worker **holds the slot** waiting for the user's `confirm_brand`, then with concurrency=1 one indecisive user **blocks the entire pipeline** until they click (or forever, if they closed the tab).
*Mitigation:* Detection must be a **fast pre-phase that releases the worker** on ambiguity: worker writes detection result, sets `awaiting_brand`, and **returns to claim other jobs**; `confirm_brand` re-queues the job for a fresh claim. Make this explicit in the worker loop — it currently reads as a single straight-through `process()`.

### TIER C — charter erosion that is invisible to a non-expert

**C1. Silent adapter rot has no detector — the scariest slow failure.**
When NARS/LF changes its page layout, extraction quietly degrades: greens that used to be VERIFIED become SINGLE_SOURCE or NOT_FOUND. The job still "succeeds," just with more reds. A non-expert has **no baseline** for what NARS coverage *should* look like, so they can't tell a broken run from a normal one — they'll ship it or over-trust the remaining greens. Golden tests exist for onboarding but there is **no continuous canary**.
*Mitigation:* Store a per-brand expected-coverage baseline (from golden/last-N runs); flag any run whose green-rate falls anomalously below it as `needs_review` with a "coverage unexpectedly low for this brand — extraction may be broken" banner, and ping Oli. This is the guardrail that keeps the charter honest over time.

**C2. Two-family independence can be fooled by syndicated content → false green.**
`source_family` = registrable domain, and green = two independent families agreeing. But many retailers ingest the **same manufacturer/distributor feed**, producing byte-identical INCI/names across "independent" domains. Widening the net worldwide (validation-net) *increases* the odds of harvesting two mirrors of one upstream source and greening on a single real source. The designs assume domain-distinct ⇒ source-independent, which is not true for syndicated data.
*Mitigation:* Treat suspiciously-identical text from different families as weak (require a minor natural variance, or down-weight exact-byte matches), and/or maintain a known-syndication cluster list that collapses to one family. At minimum, surface "2 families agreed but text was byte-identical" as a review flag rather than auto-green.

**C3. ODM_SOURCED rendered as green is affirmatively misleading.**
Flagged as a "decision" in three designs, but the risk deserves TIER-C billing: customer-**declared** values shown in the same green as independently-corroborated ones tell a non-expert "this is verified/safe" when it is exactly the opposite (uncorroborated self-report). This is the single most likely way the UI misleads someone into shipping bad data.
*Mitigation:* Distinct third color/badge ("declared", teal) — not green. Do not collapse. (This should be a decision *default*, not an open question.)

**C4. Experimental "unvalidated" warning is UI-only and does not travel with the file.**
The banner lives on the screen; the downloaded xlsx for an olaplex/aesop/svr order carries **no** "unvalidated brand" marker. Once the file circulates internally, the warning is gone and the sheet looks identical to a validated-brand output.
*Mitigation:* Stamp tier into the output file (a header cell / sheet note), same as the DRAFT stamp (A4).

**C5. Review decisions don't persist — the queue has no memory (if v1 is read-only).**
frontend defaults review to read-only v1. Then a reviewer who works 12 reds today sees the same 12 reds tomorrow with no record of "I looked at this and it's fine / I fixed it in Boozt." There is no per-cell review state, so "reds-first review queue" can't actually track review *progress*.
*Mitigation:* Even in read-only v1, allow per-cell "reviewed/accepted/needs-fix" marks (no value edit, no red→green recompute) with `reviewed_by`, persisted to a `job_cell_reviews` table. This keeps the charter (never auto-flips color) while making the queue a real worklist and feeding A4's acknowledgement gate.

### TIER D — security and operational hygiene

**D1. Signed URLs (bearer tokens for customer data) posted into Slack.**
D10 flags retention, but the sharper issue: a `sign-download` URL in a shared `#bsb-ops`/`#bsb-runs` channel is a **bearer token** — anyone in the channel (or with Slack search) can pull the customer ODM/enriched sheet for the TTL. 1h is generous.
*Mitigation:* Never put signed data URLs in **channels** — only in DMs to the requester, and prefer a **deep link to the SPA** (which re-checks RLS on click) over a raw signed URL. Short TTL (≤15 min) for any signed URL that must be sent.

**D2. No rate limiting / abuse guard on Edge Functions or uploads.**
`create-job` and `request-brand` have no throttle. A user (or a bug) can spam job creation (each a 31-min credit-burning run) or upload arbitrarily large "sample" files to `brand-samples/` (no server-side size/type cap specified).
*Mitigation:* Per-user create-job rate limit + active-job cap (the cap is mentioned but tied to the phantom `org_id` — reframe as per-user); enforce xlsx MIME + max size server-side in the Edge function and via a Storage policy.

**D3. Service-role key is a full-DB skeleton key with no rotation/segmentation plan.**
The worker holds `service_role` (bypasses all RLS) plus the Firecrawl and Slack secrets. A worker compromise or key leak = total data access + spend authority. No rotation cadence or secret-scoping is designed.
*Mitigation:* Document key rotation, keep the service-role key only on Render (never in Edge/Deno where the design already avoids it — good), and consider a scoped-down Postgres role for the worker rather than raw service_role if feasible.

**D4. User-facing failure messages are missing for the most common user error.**
A non-expert uploading a non-ODM xlsx (wrong template, a random spreadsheet) hits `EngineIngestError` → `failed` → "Oli's been pinged." The user gets an opaque failure and no guidance, and Oli gets paged for a **user error**, not a system error.
*Mitigation:* Validate the file is a plausible ODM/Specs sheet at the **detection pre-phase** and return a user-facing, actionable error ("this doesn't look like an ODM/Specs sheet — expected columns X/Y") *without* paging Oli. Reserve Oli pings for genuine engine faults.

**D5. Cache disk growth is unbounded — the volume fills and jobs fail.**
The `/cache` Render disk (the whole cost-saving mechanism) accumulates every rendered/scraped URL forever with **no eviction policy or size monitoring**. Render disks are fixed-size; when it fills, cache writes fail and jobs degrade or error.
*Mitigation:* An LRU/TTL sweep (age-based eviction of `/cache/http`) and a disk-usage alert well before full.

**D6. Regression key (`order_number`) is fragile and cross-user.**
Regression compares against the last succeeded run for the same `order_number`. But (a) detection can yield `order_number=null` (source `none`) → no key, run_history keyed on null; (b) in a team-shared model, a *different* user's unrelated upload that collides on order_number triggers a spurious regression block; (c) two genuinely different files for one order can't be distinguished from a correction.
*Mitigation:* Only run the regression gate when `order_number` is confidently detected; scope the "prior" lookup and make the comparison + override explicitly attributed; treat null-order as "no prior, no gate."

**D7. Coverage JSON has a version field but no contract test between engine and UI.**
`schema:1` is declared — good — but nothing enforces that the engine's emitted shape matches the frontend's TypeScript interface. Drift silently breaks the review queue.
*Mitigation:* A shared fixture + contract test (engine emits it, frontend parses it) in CI; bump `schema` on any change and have the UI refuse/degrade gracefully on an unknown version.

**D8. Data retention / PII policy is undefined for customer ODMs.**
Uploaded ODMs and brand-request samples (customer product/pricing, possibly contact data) sit in Storage indefinitely. No retention window, no deletion path, no GDPR posture.
*Mitigation:* Set explicit TTLs on `odm-inputs/`, `brand-samples/`; define a deletion/anonymization path; decide whether any ODM field is personal data.

## Part 2 — Decisions / inputs needed from Oli (before build)

The five designs already carry ~40 local decisions; I will not repeat those. Below are the **cross-cutting or newly-surfaced** ones the individual designs cannot resolve alone — these gate the schema and the charter.

**Must resolve before schema is written:**

1. **Tenancy model (A1):** Single org (Felina + team, domain gate only) or true multi-tenant with external customers? This determines whether `org_id` exists on every table and rewrites all RLS. The designs contradict each other; nothing can be built until this is one answer.

2. **Canonical `job_status` enum + table names (A2):** Approve one status vocabulary and one set of table names (`brands` vs `brand_registry`; `run_history` table vs view). Every design must import this, not invent its own.

3. **Progress payload shape (A3):** jsonb blob vs discrete columns, and the exact keys.

**Charter-enforcement decisions (these define whether the product is honest):**

4. **Download gating (A4):** Is "download" always-live, or gated behind an explicit reds-acknowledgement? Do we stamp the output xlsx as DRAFT + tier + red-count? (Strongly recommend: gate + stamp.)

5. **Who can disable validation, and is "Deep" self-serve (A5):** Recommend validation-off = admin-only, Deep = confirmation-gated. Confirm.

6. **ODM_SOURCED color (C3):** Distinct "declared" teal (recommended, keeps "green = corroborated" literally true) vs green. Please make this a default, not an open question.

7. **Review-decision persistence (C5):** Does read-only v1 still persist per-cell "reviewed/accepted" marks (recommended — makes the queue a real worklist without violating the color rule)?

**Cost-safety decisions (money on day one):**

8. **Ship with the Firecrawl ceiling ENABLED (B1):** Confirm a per-job default ceiling value **and** a hard global monthly kill-switch value, both enforced in `claim_next_job`/net-widening. Shipping disabled is the biggest financial risk.

9. **Retry policy for crash-class failures (B2):** Dead-letter crash/OOM jobs after 1–2 attempts (not 3) and mark worker-crashing jobs as poison → `failed`. Confirm.

**Reliability / ops decisions:**

10. **Cancellation wiring (B3) & hang watchdog (B4):** Confirm we build cooperative cancel via the progress callback and move wall-clock enforcement into the heartbeat thread (independent of progress events). These are small but load-bearing.

11. **Worker-liveness alerting (B5):** Approve an external (pg_cron/Edge) monitor for "oldest queued job age" + "worker heartbeat stale" → ping Oli. This is the one alert the worker can't send about itself.

12. **`awaiting_brand` must release the worker (B8):** Confirm detection is a fast pre-phase that frees the slot on ambiguity (not a blocking wait), given concurrency=1.

13. **Continuous canary for adapter rot (C1):** Do we run a periodic golden re-run per validated brand + per-brand coverage baseline to catch silent extraction breakage? (Recommend yes — without it the charter degrades invisibly.)

14. **Syndicated-content false-green handling (C2):** Accept byte-identical cross-family text as a review-flag rather than auto-green, and/or maintain a syndication-cluster collapse list. Confirm appetite.

**Security / compliance decisions:**

15. **Signed URLs in Slack (D1):** Confirm: DM-only, deep-link-to-SPA preferred over raw signed URL, ≤15-min TTL, never in channels.

16. **Edge-function rate limits + upload size/type caps (D2):** Confirm per-user create-job throttle + active-job cap + server-side xlsx size/MIME enforcement.

17. **Data retention / PII (D8):** Retention window for `odm-inputs`/`brand-samples`, a deletion path, and whether any ODM data is personal data under GDPR.

18. **User-error vs Oli-page split (D4):** Confirm ingest/parse failures return a user-facing actionable error and do **not** page Oli.

---

**Bottom line:** The engine, resolver, and coverage designs are deep and largely sound. The real risk is at the seams — a **tenancy contradiction, four status vocabularies, and a progress-shape mismatch** will each break the build; and the **charter is currently unenforced at the one moment that matters** (an always-live download of an unstamped sheet, with validation-off one click away and adapter rot undetectable). Resolve A1–A5 and B1 before writing schema or UI; the rest are containable during build.
